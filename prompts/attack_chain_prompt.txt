Your task is to add a concise "Attack Chain Analysis" section to an existing defensive risk report.

You receive:
1. the generated target JSON
2. the JSON quality result
3. the enhanced risk report

You must NOT:
- add new standalone risks
- remove existing risks
- change severity values
- change confidence values
- change OWASP mappings
- change STRIDE mappings
- invent components
- invent attacker capabilities
- invent tools, APIs, plugins, databases, or privileges
- provide exploit payloads
- provide jailbreak prompts
- provide bypass instructions
- provide step-by-step offensive procedures

You may only describe high-level defensive attack chains using risks and components already present in the report and JSON.

Attack chains must be:
- defensive
- conceptual
- architecture-aware
- grounded in the JSON
- concise
- non-operational

Each chain must show how one risk can enable or amplify another risk.

Use only attacker capabilities explicitly present in the JSON.

Use only components explicitly present in the JSON.

Hard rule:
- never infer privilege escalation unless explicitly supported
- never infer backend compromise
- never infer database compromise
- never infer file system compromise
- never infer code execution
- never infer admin access
- never infer external system compromise
- never infer capabilities absent from attacker limitations

If a chain requires unsupported assumptions, do not generate it.

For chat-only attackers:
Attack chains must remain bounded to:
- chatbot responses
- conversation history
- internal knowledge exposure
- misleading outputs
- availability degradation

Do not model infrastructure compromise.

If the system has no tools, APIs, plugins, or autonomous actions:
- do not include tool abuse
- do not include API abuse
- do not include plugin abuse
- do not include external system compromise
- do not include privileged action execution

Allowed chain themes:
- prompt injection leading to information disclosure
- prompt injection leading to privacy leakage
- unauthorized access to restricted knowledge base content
- conversation history exposure
- resource exhaustion leading to availability degradation
- hallucination leading to support-quality degradation

Do not describe exact prompts or operational attack steps.

Insert the Attack Chain Analysis section before "## 5. Overall Security Considerations".

## 2. Identified Assets
## 3. Attack Surface Overview
## 4. Risk Analysis
## 5. Attack Chain Analysis
## 6. Overall Security Considerations
## 7. Limitations of the Analysis

  - evidence item

  - defensive mitigation
